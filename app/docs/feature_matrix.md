# Feature Matrix (hackathon)

| Feature | Description | Status (Implemented / In progress / Planned) | Owner (Extractor/OCR/1C, Backend, Frontend, Shared) | Input/Output contracts | Definition of Done (short measurable criterion) | Risks/Notes |
|---|---|---|---|---|---|---|
| OCR from PDF | Извлечение текста из PDF через `pdf2image + pytesseract`, сохранение постраничного TXT. | Implemented | Extractor/OCR/1C | In: `data/raw/*.pdf` → Out: `data/ocr_text/*.txt` | Для тестового набора PDF сгенерированы `.txt`, в каждом есть маркеры страниц `--- page N ---`. | Качество OCR зависит от сканов, DPI и языковых пакетов Tesseract. |
| Rule-based extractor | Правила и regex для сборки структурированного JSON из OCR TXT. | Implemented | Extractor/OCR/1C | In: `data/ocr_text/*.txt` (+ опц. PDF для layout) → Out: `data/labels/*.json` | Скрипт генерирует JSON для всех входных TXT; валидные записи считаются в итоговом `valid`. | Хрупкость к новым шаблонам документов, OCR-ошибкам и верстке таблиц. |
| Schema validation | Контроль соответствия JSON схеме `draft-2020-12`. | Implemented | Shared | In: `schemas/schema.json` + `*.json` → Out: список ошибок/`valid` | `samples/valid.json` проходит, `samples/invalid.json` возвращает ожидаемые ошибки. | Возможна рассинхронизация между схемой и экстрактором при изменениях полей. |
| Labeling workflow (labels/gold) | Процесс разметки и подготовки эталонных данных для сравнения качества. | In progress | Shared | In: исходные PDF/TXT + правила из `LABELING_GUIDE.md` → Out: `data/labels/*.json`, целевой `gold`-набор | Есть гайд, согласованный gold-набор и процедура двойной проверки для спорных кейсов. | Нет формализованного пайплайна ревью/версирования gold-датасета. |
| Barcode generation | Заполнение секции `barcode` по данным документа и правилам валидации. | Planned | Extractor/OCR/1C | In: OCR/TXT поля + правила типа (`Code39/128/EAN-13`) → Out: `barcode.type/value` или `null` | Для документов с баркодом поле заполняется и проходит schema validation. | Высокий риск ложных срабатываний без специализированного barcode OCR/decoder. |
| Backend API (FastAPI) | API для запуска пайплайна, получения результатов и статусов обработки. | Planned | Backend | In: PDF/метаданные (HTTP) → Out: JSON по схеме + статус задачи | Реализованы минимум 3 endpoint: upload/process/result; OpenAPI доступен; smoke-тесты проходят. | Неопределены SLA, асинхронная обработка и формат ошибок API. |
| Frontend review UI | UI для просмотра OCR/JSON, ручной правки и подтверждения результата. | Planned | Frontend | In: JSON/API ответы → Out: подтвержденные/исправленные записи | Пользователь может открыть документ, отредактировать ключевые поля и сохранить изменения через API. | Требует стабилизации API контрактов и UX для табличных паспортов. |
| 1C REST integration | Интеграция обмена данными с 1C по REST (выгрузка/загрузка). | Planned | Extractor/OCR/1C | In: валидный JSON → Out: REST payload/статус отправки в 1C | Реализован e2e сценарий: отправка валидного JSON и получение подтверждения от тестового контура 1C. | Критичны вопросы авторизации, сетевой доступ и соответствие формату 1C. |
| Dockerized deployment | Контейнеризация сервисов и воспроизводимый запуск окружения. | Planned | Shared | In: исходники + зависимости → Out: Docker image/compose стек | `docker compose up` поднимает pipeline/API (минимальный стенд) без ручной настройки хоста. | Зависимости OCR (Tesseract/Poppler) увеличивают образ и усложняют сборку. |
| Metrics/reporting | Метрики качества извлечения и отчетность для демо/планирования. | Planned | Shared | In: labels vs gold + логи обработки → Out: precision/recall по полям, сводный отчет | Автоматически строится отчет по >=5 ключевым полям и экспортируется в JSON/CSV. | Нужен стабильный gold-набор и единая методика расчета метрик. |
